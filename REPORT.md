# Отчет по программному макету UEBA

## Назначение

Макет предназначен для обнаружения аномальной активности на ПК Windows/Linux с использованием подхода UEBA. Анализируется поведение хоста: процессы, потребление ресурсов и сетевые соединения. На этапе обучения собирается нормальный профиль работы, после чего текущие события сравниваются с профилем при помощи ML-модели.

## Архитектура

Проект разделен на клиентскую и серверную части:

1. `client/` — React-интерфейс для управления макетом.
2. `server/ueba_server/controller` — контроллеры FastAPI, аналог слоя `controller` в `lotus_game`.
3. `server/ueba_server/dto` — DTO на Pydantic для входных и выходных данных API.
4. `server/ueba_server/service` — сервисный слой: запуск UEBA-операций, jobs, проверки модели.
5. `server/ueba_server/config` — конфигурация приложения и CORS.
6. `server/ueba_prototype.collector` собирает телеметрию через `psutil`.
7. `server/ueba_prototype.features` переводит события в числовой вектор и выполняет нормализацию.
8. `server/ueba_prototype.model` обучает нейросетевой автоэнкодер и выбирает порог аномальности.
9. `server/ueba_prototype.detector` применяет сохраненную модель к текущим событиям.
10. `server/ueba_prototype.reporter` формирует отчеты и объяснения.

Backend предоставляет endpoints `/api/demo`, `/api/collect`, `/api/train`, `/api/detect/once`, `/api/detect/start`, `/api/reports`, `/api/jobs`. Длительные операции запускаются как фоновые jobs.

## Сбор данных

Команда сбора:

```bash
cd server
python -m ueba_prototype collect --duration-hours 24 --interval 5 --out data/raw.csv
```

Период сбора обучающей выборки для итогового применения должен составлять не менее 24 часов непрерывно. Это позволяет включить обычные рабочие циклы пользователя: запуск браузера, IDE, офисных приложений, фоновые обновления, типичный сетевой обмен.

## Оцифровка и нормализация

Каждое окно наблюдения преобразуется в фиксированный набор числовых признаков:

- `cpu_percent`, `memory_percent`, `swap_percent`;
- `process_count`, `user_process_count`, `new_process_count`, `thread_count`, `open_file_count`;
- `connection_count`, `established_count`, `listen_count`;
- `remote_ip_count`, `remote_port_count`, `uncommon_remote_port_count`;
- `bytes_sent_per_sec`, `bytes_recv_per_sec`, `packets_sent_per_sec`, `packets_recv_per_sec`.

Для каждого признака вычисляются среднее и стандартное отклонение на обучающей выборке. Нормализация сохраняется вместе с моделью, поэтому сохраненную модель можно использовать при последующих запусках.

## Модель ML

Используется автоэнкодер: небольшая полносвязная нейронная сеть, которая учится восстанавливать нормальные векторы поведения. Для нормальной активности ошибка восстановления мала. Если текущее поведение отличается от обучающей выборки, ошибка растет.

Порог аномальности выбирается по квантилю ошибки восстановления на обучающей выборке:

```bash
cd server
python -m ueba_prototype train --data data/raw.csv --model-dir models/default --threshold-quantile 0.995
```

## Анализ и отчеты

Запуск детектора:

```bash
cd server
python -m ueba_prototype detect --model-dir models/default --interval 5 --reports reports/anomalies.jsonl
```

Отчет содержит общий score, порог, признаки с наибольшим вкладом и объяснение. Примеры интерпретаций:

- высокий `connection_count` и `remote_ip_count`: возможное сканирование, lateral movement или C2-активность;
- высокий `bytes_sent_per_sec`: возможная эксфильтрация данных;
- высокий `uncommon_remote_port_count`: нестандартный канал связи или обход политики;
- высокий `new_process_count`: массовый запуск процессов, скриптовая активность или malware/living-off-the-land.

## Результаты тестирования

Для быстрой проверки реализован демо-сценарий:

```bash
cd server
python -m ueba_prototype demo --output-dir reports/demo
```

Сценарий генерирует нормальную синтетическую активность, обучает модель и проверяет три искусственные аномалии:

1. резкий рост количества сетевых соединений и удаленных IP;
2. передача данных на нестандартные удаленные порты;
3. массовый запуск новых процессов с высокой CPU/RAM-активностью.

Ожидаемый результат: команда сообщает о найденных аномалиях, а файл `reports/demo/demo_anomalies.csv` содержит score, порог, ключевые признаки и ИБ-объяснение.

Unit-тесты проверяют:

- корректность нормализации и обратного преобразования признаков;
- обучение модели и превышение порога для синтетической аномалии;
- генерацию объяснимого отчета.

Запуск:

```bash
cd server
python -m pytest
```

## Пользовательский интерфейс

React-клиент запускается из папки `client/`:

```bash
npm install
npm run dev
```

Интерфейс показывает статус API, наличие модели, количество отчетов, последние фоновые задачи и последние обнаруженные аномалии. Через кнопки можно запускать demo, короткий сбор, обучение, разовую проверку текущего окна и ограниченный непрерывный детектор.

## Ограничения

Макет является прототипом. Для промышленного применения нужно добавить защищенное хранение событий, централизованный сбор, контроль целостности модели, более богатую классификацию угроз, учет расписания пользователя и валидацию на реальных размеченных инцидентах.
